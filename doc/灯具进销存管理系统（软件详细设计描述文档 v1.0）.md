### 4.2.3 inventorybl模块

（1）模块概述

inventory 模块承担的需求参见需求规格说明文档功能需求及相关非功能需求。

Inventory 模块的职责及接口参见软件系统结构描述文档表 5.3.2-1 。

（2）整体结构

根据体系结构的设计，我们将系统分为展示层、业务逻辑层、数据层。每一层之间为了增加灵活性，我们会添加接口。比如展示层和业务逻辑层之间，我们添加 blservice.inventoryblservice.InventoryBLService 接口。业务逻辑层和数据层之间添加 dataservice.inventorydataservice.InventoryDataService 接口。为了隔离业务逻辑职责和逻辑控制职责，我们增加了 InventoryController，这样 InventoryController 会将对管理商品分类的业务逻辑处理委托给 Inventory 对象。InventoryBillPO 是作为库存类单据的持久化对象被添加到设计模型中去的。InventoryBill 保有库存类单据的信息，以及相应的计算合计的职责。而 InventoryList 和 InventoryLineItem 的添加是 GoodsInfo 的容器类。InventoryLineItem 保有商品变动的数量和金额等数据，以及相应的计算小计等职责。而 InventoryList 封装了关于 InventoryLineItem 的数据集合的数据结构的秘密和计算合计的职责。GoodsInfo 是Goods 提供的接口。

（2）整体结构

根据体系结构的设计，我们将系统分为展示层、业务逻辑层、数据层。每一层之间为了增加灵活性，我们会添加接口。比如展示层和业务逻辑层之间，我们添加blservice.salesblservice.SalesBLService接口。业务逻辑层和数据层之间，我们添加dataervice.salesdataservice.SalesDataService接口。为了隔离业务逻辑职责和逻辑控制职责，我们增加了PurchaseController、SalesController，这样会将PurchaseController、SalesController对进货销售的业务逻辑处理委托给Purchase、Sales对象。PurchasePO和SalesPO是作为进货单和销售单的持久化对象被添加到设计模型中去的。而PurchaseList、GoodsItem和PurchaseLineItem的添加是PurchaseInfo的容器类，SalesList、GoodsItem和SalesLineItem的添加是SalesInfo的容器类。GoodsItem保有入库商品列表商品的数据，PurchaseLineItem保有进货单的数据，封装了关于GoodsItem的数据集合的数据结构的秘密和计算总价的职责，SalesLineItem保有销售单的数据，封装了关于GoodsItem的数据集合的数据结构的秘密和计算总价的职责，PurchaseList保有所有进货单的数据，SalesItem保有所有销售单的数据。InventoryInfo、GoodsInfo、CustomerInfo和PromotionInfo都是根据依赖倒置原则，为了消除循环依赖而产生的接口。

salesbl模块各个类的职责如表所示

| 模块                 | 职责                                       |
| ------------------ | ---------------------------------------- |
| PurchaseController | 负责实现对应于制定进货单界面做需要的服务                     |
| SalesController    | 负责实现对应于制定销售单界面做需要的服务                     |
| Purchase           | 进货单的领域模型对象，拥有单据编号、供应商、仓库、操作员、入库商品列表、备注和总额合计，可以帮助完成制定进货单界面所需要的服务 |
| Sales              | 销售单的领域模型对象，拥有单据编号、客户、业务员、操作员、仓库、出库商品清单、折让前总额、折让、使用代金券金额、折让后总额和备注，可以帮助完成制定销售单界面所需要的服务 |
| PurchaseList       | 负责实现对应于展示进货单、进货退货单列表做需要的服务               |
| SalesList          | 负责实现对应于展示销售单、销售退货单列表做需要的服务               |
| PurchaseLineItem   | 负责实现对应于进货单、进货退货单具体内容做需要的服务               |
| SalesLineItem      | 负责实现对应于销售单、销售退货单具体内容做需要的服务               |
| GoodsItem          | 负责实现对应于商品清单做需要的服务                        |

### 4.2.6 accountbl模块

（1）模块概述

accountbl 模块承担的需求参见需求规格说明文档功能需求及相关非功能需求

accountbl 模块的职责及接口参见软件系统体系结构描述文档表5.3.2-4

（2）整体结构 

根据体系结构的设计，我们将系统分为展示层、业务逻辑层、数据层。每一层之间为了增加灵活性，我们会添加接口。在展示层和业务逻辑层之间，我们添加AccountBLService接口。在业务逻辑层和数据层之间，我们添加AccountDataService接口。为了隔离业务逻辑职责和逻辑控制职责，我们增加了AccoutController，这样AccountController会将对银行账户的业务逻辑处理委托给Account对象。AccountPO是作为银行账户的持久化对象被添加到设计模型中去的。

### 4.2.7 financebl模块 

（1）模块概述

financebl 模块承担的需求参见需求规格说明文档功能需求及相关非功能需求

financebl 模块的职责及接口参见软件系统体系结构描述文档表5.3.2-5

（2）整体结构 

根据体系结构的设计，我们将系统分为展示层、业务逻辑层、数据层。每一层之间为了增加灵活性，我们会添加接口。在展示层和业务逻辑层之间，我们添加FinanceBLService接口。在业务逻辑层和数据层之间，我们添加FinanceDataService接口。为了隔离业务逻辑职责和逻辑控制职责，我们增加了FinanceController，这样FinanceController会将对财务单据和财务报表的业务逻辑处理委托给Finance对象。AccountBillPO和CashBillPO是作为收付款单和现金费用单的持久化对象被添加到设计模型中去的。AccountBillItemPO和CashBillItemPO是作为转账列表和条目清单的持久化对象被添加到设计模型中去的。AccountBillItem保有银行账户名称、转账金额和备注的数据。CashBillItem保有条目名、金额、备注的数据。AccountBill负责管理AccountBillItem，并有计算总额汇总的职责。CashBillItem负责管理CashBillItem，并有计算总额汇总的职责。SalesDetails负责管理SalesDetailsItem，并有生成和导出销售明细表的职责。Profit负责生成和导出经营情况表。DocumentDetails负责生成和导出经营历程表。UserInfo、AccountInfo、CustomerInfo、InventoryInfo、SalesInfo都是根据依赖倒置原则，为了消除循环依赖而产生的接口。

### 4.2.10 promotionbl模块

（1）模块概述

promotionbl模块承担的需求参见需求规格说明文档功能需求及相关非功能需求。

promotionbl模块的职责及接口参见软件系统结构描述文档。

（2）整体结构

根据体系结构的设计，我们将系统分为展示层、业务逻辑层、数据层。每一层之间为了增加灵活性，我们会添加接口。比如展示层和业务逻辑层之间，我们添加blservice.promotionbl.PromotionBLService接口。业务逻辑层和数据层之间添加dataservice.promotiondataservice.PromotionDataService接口。为了隔离业务逻辑职责和逻辑控制职责，我们增加了PromotionController，这样PromotionContorller会将对促销策略的业务逻辑处理委托给Promotion对象。PromotionTotalPO、PromotionCustomerPO和PromotionBargainPO是作为促销策略的持久化对象被添加到设计模型中去的。而PromotionList和PromotionListItem的添加是GoodsInfo的容器类。PromotionListItem保有促销策略的类型、对象、促销商品、赠品、代金券、折让等数据。而PromotionList封装了关于PromotionListItem的数据集合的数据结构的秘密。GoodsInfo是根据依赖倒置原则，为了消除循环依赖而产生的接口。
